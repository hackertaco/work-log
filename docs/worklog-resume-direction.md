# Work Log -> Resume Direction

## Why this needs to change

현재 `Living Resume`는 업무로그에서 신호를 모으고는 있지만, 사용자가 기대하는 방식으로 이력서까지 자연스럽게 이어지지 않는다.

핵심 문제는 `추출`은 있는데 `승격`이 없다는 점이다.

- 업무로그에는 사건과 작업 기록이 쌓인다.
- 현재 이력서 쪽은 키워드, 축, 강점 신호를 보여주지만 이것이 곧바로 "이 사람이 어떤 문제를 반복해서 잘 푸는가"로 이어지지 않는다.
- 그래서 사용자는 업무로그와 이력서가 같은 데이터를 보고 있다는 느낌을 받지 못한다.

좋은 이력서는 최근 작업의 나열이 아니라, 반복적으로 증명된 가치의 압축본이어야 한다. 따라서 이 기능은 `문장 추출기`가 아니라 `증거를 상위 개념으로 승격하는 시스템`이 되어야 한다.

## Product thesis

`work-log`는 업무로그를 읽어 다음 세 가지를 자동으로 도출해야 한다.

- `핵심 프로젝트`
- `반복 강점`
- `상위 축`

그리고 이 셋이 서로 연결되어 이력서에 반영되어야 한다.

권장 관계는 다음과 같다.

1. `업무로그`는 원본 증거다.
2. 여러 업무로그를 묶어 `핵심 프로젝트`를 만든다.
3. 여러 프로젝트와 로그에서 반복 검증된 패턴을 `강점`으로 승격한다.
4. 강점과 프로젝트를 묶어 "어떤 문제를 푸는 사람인가"를 설명하는 `축`을 만든다.
5. 이력서 문장과 카드들은 이 구조에서 파생된다.

즉 최종 구조는 `축 -> 강점 -> 핵심 프로젝트 -> 원본 업무로그`가 되어야 한다.

## What good looks like

사용자가 업무로그를 꾸준히 쓰면, 이력서를 열었을 때 아래가 이미 어느 정도 채워져 있어야 한다.

- 지금 가장 대표적인 프로젝트가 무엇인지
- 최근 반복해서 드러난 강점이 무엇인지
- 본인을 어떤 한두 문장으로 설명할 수 있는지

그리고 사용자는 대부분을 새로 쓰는 대신, `승인`, `수정`, `제외` 정도만 하면 되어야 한다.

이 기능이 잘 된 상태의 판단 기준:

- 사용자가 자동 제안의 대부분을 큰 수정 없이 수용한다.
- 핵심 프로젝트가 단일 로그가 아니라 여러 날짜의 작업 흐름을 반영한다.
- 강점이 키워드 나열이 아니라 행동 패턴과 결과로 읽힌다.
- 축이 `Backend / Cloud / Data` 같은 태그 묶음이 아니라 서사로 읽힌다.

## Recommended model

### 1. Evidence layer

업무로그에서 먼저 아래 단위의 증거를 뽑는다.

- 문제
- 내가 한 행동
- 결과 또는 영향
- 사용 기술
- 관련 프로젝트 후보
- 근거가 된 날짜와 로그

여기서 중요한 것은 키워드보다 `문제-행동-결과` 구조다.

### 2. Promotion layer

증거를 바로 이력서 문장으로 바꾸지 말고, 먼저 상위 개념으로 승격한다.

- `핵심 프로젝트`
  - 여러 로그에 걸쳐 반복 등장
  - 목표나 문제 맥락이 존재
  - 의미 있는 결과 또는 사용자 가치가 있음
- `강점`
  - 서로 다른 프로젝트나 로그에서 반복 검증됨
  - 예: 운영 안정성, 제품 판단력, 도구화
- `축`
  - 강점과 프로젝트를 묶는 상위 포지셔닝
  - 예: 운영 복잡도를 제품 흐름으로 바꾸는 엔지니어

### 3. Resume layer

이력서는 아래 질문에 답하는 뷰가 되어야 한다.

- 이 사람의 대표 프로젝트는 무엇인가
- 어떤 강점이 반복적으로 증명되었는가
- 그래서 어떤 역할에 적합한가

이때 각 항목은 원본 업무로그 근거를 역추적할 수 있어야 한다.

## Implications for the current codebase

현재 구현은 대체로 `키워드 -> 축` 중심이다.

- [resumeRecluster.mjs](/Users/seungahjung/Documents/company-code/work-log/src/lib/resumeRecluster.mjs)
  - 미분류 키워드 비율을 기준으로 재클러스터링한다.
  - 이 방식은 "무슨 단어가 많이 나왔는가"는 잡지만 "무슨 프로젝트가 성장했고 어떤 강점이 반복 검증되었는가"는 잘 못 잡는다.
- [DisplayAxesView.jsx](/Users/seungahjung/Documents/company-code/work-log/frontend/src/components/resume/DisplayAxesView.jsx)
  - 프로필 분석이 키워드 커버리지와 축 재편성 중심이다.
  - 사용자가 원하는 것은 분석 정확도보다 "이력서로 이어지는 의미"다.
- [WorkLogPage.jsx](/Users/seungahjung/Documents/company-code/work-log/frontend/src/pages/WorkLogPage.jsx)
  - 현재 스냅샷은 강점/기술/작업 성향 요약이 중심이다.
  - 이 요약이 `핵심 프로젝트`나 `이력서 서사`까지 올라가지 못해 연결감이 약하다.

따라서 다음 전환이 필요하다.

- `키워드 클러스터링`을 핵심 모델로 두지 않는다.
- `핵심 프로젝트 추출`을 1급 개념으로 올린다.
- `강점`은 키워드 묶음이 아니라 프로젝트와 로그를 가로지르는 반복 패턴으로 계산한다.
- `축`은 키워드 묶음이 아니라 강점과 프로젝트를 설명하는 상위 문장으로 만든다.

## Suggested UX

업무로그 화면에서는:

- 오늘의 작업 요약
- 누적되는 프로젝트 후보
- 최근 강점 신호
- "이 내용이 어떤 핵심 프로젝트로 자라고 있는지"를 보여준다

이력서 화면에서는:

- 대표 축 2~3개
- 각 축 아래 반복 강점
- 각 강점 아래 대표 프로젝트
- 각 프로젝트에 연결된 근거 로그

사용자 액션은 최소화한다.

- 승격
- 이름 수정
- 제외
- 대표 프로젝트로 고정

직접 키워드를 이리저리 옮기는 UI는 중심 경험이 아니어야 한다.

## Near-term build direction

다음 순서가 현실적이다.

1. 현재 `키워드/축` 흐름은 유지하되 중심 UI에서 비중을 낮춘다.
2. 업무로그에서 `핵심 프로젝트 후보`를 만드는 집계 레이어를 추가한다.
3. 프로젝트 후보들을 바탕으로 `반복 강점`을 재계산한다.
4. 강점과 프로젝트를 묶어 `축`을 다시 생성한다.
5. 이력서 문장은 위 구조를 바탕으로 생성하고, 각 문장에 근거 로그를 연결한다.

## Success criteria

- 업무로그를 일정량 쌓은 뒤 이력서를 열면 대표 프로젝트, 강점, 축이 자동으로 채워진다.
- 사용자는 자동 제안의 대부분을 큰 수정 없이 수용할 수 있다.
- 각 항목은 어떤 업무로그에서 유래했는지 확인할 수 있다.
- 이력서가 `키워드 모음`이 아니라 `프로젝트와 강점이 연결된 서사`로 읽힌다.
